# DOCX Merger

Компактное кроссплатформенное desktop-приложение на Python для объединения нескольких `.docx`-файлов в один итоговый документ с максимально возможным сохранением исходного форматирования на бесплатном стеке `docxcompose` + `python-docx`.

Приложение использует:

- `Python 3.12` или `3.13`
- `PySide6` для графического интерфейса
- `docxcompose` для объединения документов
- `python-docx` для подготовки документов и режимов разрыва
- `QSettings` для хранения пользовательских настроек

Важно: `Python 3.14` сейчас не подходит для этого проекта, потому что доступные сборки `PySide6` не гарантируют установку и работу на этой версии Python.

## Возможности

- добавление `.docx`-файлов через диалог выбора и drag-and-drop;
- изменение порядка файлов кнопками и перетаскиванием внутри списка;
- удаление одного или нескольких выбранных файлов;
- выбор папки сохранения;
- задание имени итогового файла;
- выбор режима объединения:
  - разрыв страницы;
  - разрыв раздела;
  - без разрыва;
- асинхронное выполнение объединения в отдельном потоке;
- progress bar, процент выполнения и приблизительное оставшееся время;
- отмена операции;
- встроенный журнал событий;
- открытие итогового файла и папки результата;
- сохранение пользовательских настроек между запусками.

## Как пользоваться

1. Добавьте минимум два `.docx`-файла кнопкой `Добавить файлы` или перетащите их в окно.
2. Измените порядок файлов кнопками `Вверх` / `Вниз` или drag-and-drop внутри списка.
3. Выберите папку сохранения.
4. Укажите имя итогового файла.
5. Выберите режим объединения.
6. Нажмите `Объединить`.
7. При необходимости используйте `Отменить`.

Дополнительно доступны:

- журнал событий;
- открытие итогового файла;
- открытие папки результата;
- показ итогового файла в Finder или Проводнике;
- окна `О программе` и `Инструкция`.

## Известные ограничения

- Поддерживаются только файлы `.docx`.
- Итог объединения максимально близок к бесплатному варианту вставки содержимого, но абсолютной идентичности ручной вставке в Microsoft Word гарантировать нельзя.
- Некоторые документы со сложной структурой Word могут вести себя по-разному из-за ограничений `docxcompose` и `python-docx`:
  - колонтитулы;
  - разрывы секций;
  - сквозная нумерация страниц;
  - сложные стили с одинаковыми именами;
  - поля Word, комментарии, оглавления, сноски.
- Колонтитулы и часть секционных настроек в первую очередь наследуются от первого документа, это известная особенность `docxcompose`.
- Оценка оставшегося времени приблизительная и зависит от количества и сложности документов.
- Отмена операции кооперативная: она выполняется в безопасной точке, а не мгновенно посреди внутренней операции библиотеки.
- На момент подготовки проекта установка на `Python 3.14` не поддерживается практическим набором доступных wheels для `PySide6`.

## Особенности кроссплатформенной работы на macOS и Windows

### Что реализовано одинаково

- интерфейс на `PySide6`;
- список файлов и drag-and-drop;
- изменение порядка документов;
- объединение документов через `docxcompose` и `python-docx`;
- фоновые потоки через `QThread`;
- сохранение настроек через `QSettings`;
- текстовый лог и статусные сообщения;
- валидация файлов и имени результата.

### Что может вести себя немного по-разному

- внешний вид нативных диалогов выбора файлов и папок;
- поведение системного меню окна;
- способ открытия файла, папки и показа файла в файловом менеджере;
- фактическое место хранения настроек `QSettings`:
  - на macOS это нативное хранилище системы;
  - на Windows это стандартное хранилище Windows для приложений;
- допустимые символы в имени файла результата;
- оформление системных шрифтов и элементов окна.

### Где инкапсулированы платформенные различия

Платформенные различия вынесены в модуль:

- `app/platform_utils.py`

Там находятся:

- определение текущей платформы;
- нормализация путей;
- открытие папки;
- открытие файла;
- показ файла в Finder или Проводнике;
- часть платформенной валидации имен файлов.

## Где менять название, версию, автора и GitHub

Основные runtime-метаданные собраны в одном файле:

- `app/app_info.py`

Именно там меняются:

- название приложения;
- версия приложения;
- автор;
- ссылка на GitHub;
- имя и домен организации для `QApplication`.

Дополнительно версию пакета нужно синхронизировать в:

- `pyproject.toml`

## Краткое описание файлов

- `main.py` — точка входа, инициализация `QApplication`, логирования и окна.
- `app/app_info.py` — название, версия, автор, GitHub и организационные метаданные.
- `app/ui.py` — главное окно приложения, меню, список файлов, статусы, лог, действия пользователя.
- `app/merger.py` — ядро объединения `.docx` через `docxcompose` и `python-docx`.
- `app/worker.py` — выполнение объединения в отдельном потоке, прогресс, отмена, сигналы.
- `app/settings_manager.py` — сохранение и восстановление настроек через `QSettings`.
- `app/logger.py` — внутренний логгер приложения, вывод в файл, консоль и интерфейс.
- `app/platform_utils.py` — кроссплатформенные функции для macOS и Windows.
- `app/dialogs.py` — диалоги выбора файлов и папок, служебные окна `О программе` и `Инструкция`.
- `app/utils.py` — общие вспомогательные функции валидации и преобразования ошибок.
